# Project Context: Scoretopia

> ⚠️ Some decisions below are still open (Discord library, persistence layer).
> They're flagged inline. The Planner should raise them explicitly the first
> time a task touches that area, rather than silently picking one.

## 🎯 Overview
Scoretopia helps a group of friends who play the mobile 4X game **Polytopia**
track wins and losses across their games. It lives in a **Discord channel**
the group already uses to coordinate games — players interact with the bot
there rather than through a separate app or website.

**Phase 1 is manual**: players (or a designated recorder) report game results
to the bot via Discord commands/messages, and the bot stores and surfaces
standings (e.g. win/loss records, leaderboards) on request. There is no
integration with the Polytopia game client itself yet.

**Future phases are open-ended** — the roadmap is intentionally vague right
now because it depends on ideas that come up as Phase 1 is used in practice.
Do not over-build for hypothetical automation (e.g. screen-scraping the game,
parsing screenshots, or a Polytopia API integration if one ever exists) —
design Phase 1 so it's *not painful to extend*, but don't build the extension
points speculatively.

## 🏗️ Core Architecture
- **Single Discord server (guild), single channel** is the assumed scope for
  now — no multi-tenant / multi-server design needed unless that changes.
- **Players** are Discord users; the bot needs some way to map a Discord user
  to a "Polytopia player" identity for the record-keeping (a person's Discord
  handle and their in-game name may differ).
- **Game result recording is the core primitive**: who played, who won/lost,
  and (eventually) enough metadata to support whatever stats get asked for
  later (date, map/settings, participant list). Get this data model right
  early — it's the thing every future feature builds on.
- Manual Phase 1 workflow implies **trust in whoever reports a result** — no
  need for elaborate verification/dispute handling yet, but don't design
  anything that would make adding a "confirm this result" step painful later.

## 🧱 Stack
- **Language**: Python.
- **Discord library**: **Not yet decided.** `discord.py` is the natural
  default (actively maintained, async, slash-command support) — confirm with
  the user before scaffolding the bot rather than assuming.
- **Persistence**: **Not yet decided.** Candidates discussed: SQLite (simple,
  file-based, queryable — a good fit for a small group) vs. a plain JSON/YAML
  file (simplest, but no query capability or concurrency safety). Confirm
  before writing any storage code.
- **Tests**: `pytest` (wired into `scripts/test.sh`).
- **Lint**: `ruff` (wired into `scripts/lint.sh`); add `mypy` later if the
  project wants static typing enforced.

## 📂 Key Areas of the Codebase

Update this section once that structure exists so future planning sessions

## 🔁 Development Workflow (Planner → Worker)
This project separates planning from execution:
- **Planning**: An architect/planner session (Cursor `planner.mdc` rule) turns
  a rough idea into role-typed task files in `docs/tasks/ready/`. Human
  approves before anything enters the queue.
- **Execution**: A worker session picks up tasks via the `scoretopia-tdd` MCP
  server, which enforces a TDD loop (test → implement → refactor) and runs
  `scripts/lint.sh` + `scripts/test.sh` at each phase. The worker hands off git
  commands; the human commits, opens a PR, and merges.

## ⚠️ Known Gotchas
- **Discord identity vs. Polytopia identity** — a player's Discord handle and
  their in-game Polytopia name are different things; don't assume they match
  or that one can be derived from the other.
- **No verification step in Phase 1** — whoever reports a result is trusted.
  Don't add dispute/confirmation flows unless asked; do keep the data model
  from actively fighting a future "unconfirmed result" state.
- **Scope creep risk is high here** — "automate more later" is an explicit
  goal, which makes it tempting to over-engineer Phase 1 for imagined future
  automation. Prefer the simplest thing that solves the immediate ask.
